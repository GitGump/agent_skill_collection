# 我要吐槽 —— 深度现象分析与风趣解读

## 任务目标
本 Skill 用于：深度剖析社会现象、行业趋势、文化热点，以第三人称视角创作兼具专业性与可读性的分析文章，或以风趣幽默但克制的方式进行吐槽解读。

能力包含：
1. 挖掘现象背后的深层机制与本质
2. 创作兼具思考深度与活人感的分析文章
3. 提供专业视角的深度解读或风趣克制的吐槽
4. 完成适合公众号展示的排版优化
5. 生成可直接复制到公众号的 HTML 文件

触发条件：
- 用户说「我想吐槽一下XXX」
- 用户说「帮我分析一下XXX这个现象」
- 用户提供一个社会现象、行业趋势、文化热点
- 用户说「写一篇现象分析」「解读一下这个话题」
- 用户提到职场内卷、消费主义、互联网乱象、社会心理、代际差异、城市生活等主题

## 操作步骤

### 标准流程

**步骤1：现象分析与框架构建**

基于用户提供的话题/现象，智能体将：

1. **判断话题适配性**：
   - 是否具有普遍性：能否引起广泛共鸣
   - 是否有深度可挖：表层现象下是否有机制、心理、文化层面的原因
   - 是否适合解读或吐槽：有些话题适合严肃分析，有些适合风趣吐槽，有些可以两者结合

2. **确定创作基调**：
   - **严肃分析型**：适合社会问题、行业趋势、文化现象等需要深度解读的话题
     - 基调：专业、冷静、有洞察力
     - 语言：逻辑清晰，用词精准，避免过度口语化
   - **风趣吐槽型**：适合生活趣事、消费怪象、互联网乱象等可以轻松调侃的话题
     - 基调：幽默、犀利、但不刻薄
     - 语言：活泼、机智、有梗但不过度
   - **混合型**：适合既有深度又有槽点的话题
     - 先吐槽吸引注意，后分析提供价值
     - 或先分析建立权威，后吐槽拉近距离

3. **构建分析框架**：
   - **现象描述**：具体、生动、有画面感（但不虚构人物故事）
   - **表层分析**：显而易见的原因，大众普遍认知
   - **深层挖掘**：机制、心理、文化、历史、经济等维度
   - **本质提炼**：这个现象到底反映了什么
   - **延伸思考**：可能的发展趋势、应对方式、价值判断

**选题参考**：见 [references/topic-guide.md](references/topic-guide.md)

**步骤2：文章创作**

以「现象分析」风格创作文章：

**写作风格核心特征**：
- **叙述视角**：第三人称，客观观察者视角，但要有温度和立场
- **语言质地**：
  - 分析型：专业但不晦涩，有逻辑但不刻板
  - 吐槽型：风趣但不刻薄，幽默但不油滑
  - 通用原则：口语化适度，书面化有温度
- **叙事节奏**：张弛有度，有轻有重，避免全程高能或全程平铺
- **思考姿态**：有观点但不强加，有态度但不教条
- **情感原则**：引发共鸣而非煽情，提供洞见而非鸡汤
- **禁止词汇**：禁用「值得深思」「发人深省」「令人唏嘘」「不禁感慨」等套路表达
- **结尾处理**：
  - 分析型：留下延伸思考的空间，不强行给答案
  - 吐槽型：以轻松但不轻浮的方式收尾，给人回味
- **可读性**：避免学术论文式的枯燥，让复杂观点变得易于消化
- **共鸣感**：从读者的真实体验出发，让他们看到自己的生活

**代表性叙事技法**：
- 开头用一个具体的场景、现象或数据切入（不虚构人物，但可以描述典型情境）
- 用类比和隐喻让抽象概念具象化（但避免过度使用）
- 穿插具体案例（可以是真实事件，也可以是典型场景描述）
- 避免连续议论，适当加入轻松笔触或反问句调节节奏
- 段落之间有逻辑递进，不是简单的并列和罗列
- **文章标题：简洁有力，像一个问题或一个判断，不用感叹号**
- **要有专业深度**：不能只是情绪宣泄，要有知识储备和思考能力支撑

**标题风格**：

| 类型 | 公式 | 示例 |
|------|------|------|
| 分析型 | [现象/问题] + [判断] | 《为什么年轻人不爱接电话》 |
| 吐槽型 | [现象] + [讽刺感] | 《办公室里的表演性忙碌》 |
| 混合型 | [情绪] + [反问] | 《我们为什么对朋友圈的精致生活感到疲惫》 |

1. **生成文章结构**：
   - 字数：**2000～4000 字**（根据话题深度灵活调整）
   - 分 5～8 个段落，每个段落聚焦一个分析维度或论述角度
   - 结构清晰但不刻板，可使用小标题分割（但不滥用）

2. **撰写正文内容**：
   - 开篇：从一个具体现象、数据或场景切入，3～5 句内建立读者共鸣
   - 展开：层层递进的分析，从表层到深层
     - 第一层：现象本身是什么，有多普遍
     - 第二层：显而易见的原因，大众怎么看
     - 第三层：深层的机制、心理、文化原因
     - 第四层：这个现象反映了什么本质问题
   - 论证：用具体案例、数据或类比支撑观点（不虚构人物故事，但可引用真实事件或描述典型情境）
   - 转折：指出可能的反驳意见，或现象的复杂性（避免观点过于单一）
   - 收尾：留有余韵的思考，不强行总结升华

3. **优化润色**：
   - 检查是否有「鸡汤文」套路痕迹（如「其实我们都在」「真正重要的是」）
   - 检查分析是否有深度：是否只是现象罗列而没有本质挖掘
   - 检查吐槽是否有分寸：是否变成了纯粹的抱怨或攻击
   - 确认观点是否过于绝对：好的分析应该承认复杂性
   - 语言节奏是否有变化：避免全程一个调子，要有轻重缓急
   - 检查可读性：复杂概念是否解释清楚，专业术语是否有铺垫
   - 检查共鸣感：是否从读者真实体验出发，而不是高高在上地指点

4. **生成标题与文件名**（文章完成后）：
   - 基于全文内容，提炼核心观点或核心问题
   - 标题风格：**简洁、有力、有观点**
   - **文件名必须带时间戳**：格式为 `{标题}-{YYYYMMDDHHmm}`
     - 例如：`为什么年轻人不爱接电话-202605101105`
     - 时间戳取当前系统时间（精确到分钟）
     - **目的**：同一主题多次生成的文章不互相覆盖
   - 生成 3～5 个备选标题，优先选最简洁有力的那个
   - 避免：感叹号、标题党词汇（「震惊」「绝了」「真相」）

**标题生成参考**：见 [references/topic-guide.md](references/topic-guide.md)

**步骤3：事实信息筛查【必须执行】**

> ⚠️ 这是文章输出前的强制质检步骤，不得跳过。

#### 筛查原则

1. **时间基准**：获取当前对话时间戳，作为数据时效判断的基准。所有数据信息必须是当前时间或能获取最新数据的最近时间（精确到月份），禁止引用明显过时的数据。
2. **必须验证**：所有事实性内容必须经过准确性核查，不能凭记忆填写。
3. **来源对比**：如有疑问，使用 web_search 进行多源交叉验证（至少2个不同来源）。

#### 筛查范围

以下类型信息必须逐项核查：

| 信息类型 | 验证要点 | 处理方式 |
|---------|---------|---------|
| **时间信息** | 事件发生年份、政策发布时间、数据统计周期 | 与官方信息比对，数字有误则更正 |
| **数据信息** | 统计数字、百分比、金额、人数 | 获取当前月份最新数据，标注数据来源和时间 |
| **第三方引用** | 专家观点、报道内容、研究报告结论 | 确认引用的实际来源，无法确认则改为「据报道」「有分析认为」 |
| **政策/法规** | 政策名称、生效时间、具体条款 | 与官方发布渠道核对，避免引用已废止或修订的内容 |
| **个人观点** | 公众人物表达的个人立场、态度 | 确认该观点是本人公开表达，或明确标注为推测/分析 |
| **机构信息** | 公司名称、组织架构、行业排名 | 与官网或权威媒体报道比对 |

#### 筛查流程

```
1. 列出文章中所有事实性表述（含数字、时间、人名、机构名、政策名称）
2. 逐条判断可信度：
   - 高度确信 → 保留
   - 存疑 → 使用 web_search 验证
   - 无法确认 → 删除或加「据说/据报道」
3. 对所有数字（统计值/金额/百分比）进行时效检查
4. 检查是否存在「流传甚广但未经证实」的内容
```

#### 筛查结果输出

在文章正文之后，单独输出筛查报告，格式如下：

```
---

## 📋 事实信息筛查报告

**核查时间**：YYYY年MM月DD日 HH:MM（对话时间戳）
**核查条目**：共 N 项

| 序号 | 信息内容 | 核查状态 | 来源/说明 |
|-----|---------|---------|---------|
| 1 | 某政策于2024年X月生效 | ✅ 已核实 | 政府官网 |
| 2 | 某统计数据显示XX% | ✅ 已核实，截至YYYY-MM | 某某统计局 |
| 3 | 某专家曾表示XXX | ⚠️ 未能确认 | 改为「有分析认为」 |
| 4 | 某事件发生于2023年 | ✅ 已核实 | 多家媒体报道 |

**修改说明**：
- 第3条：无法确认原始来源，已将表述调整为「有分析认为」
```

**步骤4：排版优化**

我要吐槽专属排版风格：

1. **整体基调**：
   - 分析型：简洁、现代、专业感，适当留白
   - 吐槽型：轻松、活泼、有节奏感，不过度设计
   - 主色调：深灰/黑色（分析型），可适当加入品牌色点缀

2. **结构设计**：
   - 可使用小标题分割论述板块（但不滥用）
   - 小标题风格：
     - 分析型：简洁的数字序号（一、二、三）或概念性标题
     - 吐槽型：有梗的小标题（但不过度标题党）
   - 金句或核心观点可用引用样式（`>` 块）突出

3. **细节处理**：
   - 重要概念可加粗，但不滥用
   - 数据或关键信息可用引用样式
   - 不使用任何分隔线（`---` / `———`）

4. **结尾互动**：
   - 文章结尾加一段轻柔的引导语
   - 示例：
     - 分析型：「你有没有观察过这个现象？欢迎分享你的看法。」
     - 吐槽型：「你怎么看这个现象？来评论区聊聊。」

**步骤6：质量检查与最终交付**

1. **内容检查**：
   - 分析是否有深度：是否只是现象罗列而无本质挖掘
   - 吐槽是否有分寸：是否变成了情绪宣泄或人身攻击
   - 观点是否过于绝对：好的分析应该承认现象的复杂性
   - 语言是否流畅：是否过度使用术语或过度口语化
   - 可读性如何：复杂概念是否解释清楚
   - 共鸣感如何：是否从读者真实体验出发
   - 有无套路痕迹：检查是否使用了「值得深思」「发人深省」等模板词汇
   - 逻辑是否清晰：段落之间是否有递进关系，而非简单并列

2. **排版检查**：
   - 有无分隔线？（须删除）
   - 小标题数量是否合适？（分析型可适当多，吐槽型可适当少）
   - 段落长度是否适中？

3. **HTML 检查**：
   - 在浏览器预览，样式是否正确显示？
   - HTML 文件大小是否合理（建议 <5MB）？

4. **事实筛查检查**：
   - ✅ 是否已执行步骤3的事实信息筛查？
   - ✅ 筛查报告是否已附在文章末尾？
   - ✅ 存疑条目是否已标注或修正？

**最终产物清单**：
- ✅ `[文章标题-时间戳].html` — 可直接复制到公众号编辑器的 HTML 文件
- ✅ `[文章标题-时间戳].md` — Markdown 源文件（含事实筛查报告）
- ✅ 若产物不止一个，所有文件统一保存在以 `{文章标题-时间戳}` 命名的子文件夹中

## 注意事项
- **不虚构人物故事**：这与「真实故事计划」的核心区别。可以有典型情境描述，但不能虚构具体人物的经历和对话
- **不情绪宣泄**：吐槽要有智慧，不是抱怨；分析要有深度，不是情绪化表达
- **不只罗列现象**：必须挖掘机制、心理、文化等深层原因
- **不强行升华**：结尾留有余韵，不强行给出人生道理
- **承认复杂性**：好的分析应该指出现象的多面性，而非单一归因
- **从读者出发**：引发共鸣的前提是理解读者的真实体验
- 排版不使用任何分隔线
- 图片标签格式：外部 URL 用 `![描述](https://URL)`，本地图片用 `![描述](./images/名.png)`
- 充分利用分析能力，以思考深度和专业性取胜，避免套路化写作
- **事实筛查为强制项**：每篇文章输出前必须完成事实信息核查并附筛查报告

## 使用示例

**示例1：分析型**
- 用户输入：「帮我分析一下为什么年轻人不爱接电话」
- 执行：从通讯方式变迁、社交焦虑、工作边界模糊等角度层层剖析
- 关键处理：避免简单归因，指出现象的复杂性；不虚构人物故事，但可描述典型情境

**示例2：吐槽型**
- 用户输入：「吐槽一下办公室里的表演性忙碌」
- 执行：用风趣但不刻薄的笔触描绘现象，分析其背后的职场文化机制
- 关键处理：幽默有分寸，不过度讽刺；仍要有思考深度，不只是调侃

**示例3：混合型**
- 用户输入：「分析一下当代人的养生焦虑」
- 执行：先吐槽朋克养生等现象，再深入分析焦虑的社会根源
- 关键处理：吐槽吸引注意，分析提供价值；两者平衡，不偏废其一
