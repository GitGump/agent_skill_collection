# 真实故事计划 —— 微信公众号非虚构写作助手

## 任务目标
本 Skill 用于：模仿「真实故事计划」风格，快速生成高质量非虚构类微信公众号文章，从选题策划、事实核查、故事创作到排版发布的一站式服务。

能力包含：
1. 挖掘符合「真实故事计划」调性的选题
2. 以第一人称叙事为主，创作 **2000～5000 字**的非虚构风格公众号文章
3. **对素材中所有事实性内容进行准确性验证（必须执行）**
4. 自动完成文章排版（适合公众号展示，克制、留白、沉浸）
5. 生成可直接复制到公众号的 HTML 文件
6. 提供发布操作指引

触发条件：
- 用户说「帮我写一篇真实故事计划风格的文章」
- 用户提供一个真实故事线索/采访素材/生活观察
- 用户说「写一篇非虚构」「写一个普通人的故事」「写底层叙事」
- 用户提到职场、情感、边缘群体、乡土中国、城市漂泊、生死、疾病、犯罪、小众职业等主题

## 操作步骤

### 标准流程

**步骤1：选题与采访素材梳理**

基于用户提供的线索/主题，智能体将：

1. **判断主题适配性**：
   - 是否属于真实故事计划核心选题范围：职场、情感、边缘群体、乡土中国、城市漂泊、生死、疾病、犯罪、小众职业
   - 评估故事中是否有真实的人物、具体的处境、情感的流动
   - 确认：是否有「人性的挣扎与微光」——这是最核心的价值判断

2. **提炼核心叙事角度**：
   - 这个故事从谁的视角讲？（第一人称叙述者）
   - 这个人面对的核心困境是什么？
   - 故事最打动人的细节/时刻在哪里？
   - 不追求猎奇，而是挖掘**普通处境中的非凡体验**

3. **建立故事结构骨架**：
   - 开场：一个具体的场景、细节或动作，直接抛入故事中
   - 背景铺垫：主人公是谁，她/他从哪里来
   - 核心事件：改变发生的那个节点
   - 情感流动：内心世界的挣扎与选择
   - 余韵结尾：不说教，不升华，留白给读者

**选题参考**：见 [references/story-topic-guide.md](references/story-topic-guide.md)

---

**步骤2：事实信息核查（必须执行，不可跳过）**

在正式写作前，对用户素材及计划引用的所有事实性内容进行系统性核查：

**核查范围**（涵盖以下所有类别）：
- **时间信息**：事件发生时间、人物经历年份、政策/法规生效时间等
- **数据信息**：统计数字、比例、排名、调查结果等——**数据必须基于当前时间（取对话时间戳）或可获取的最新月份数据，不使用过时数据**
- **第三方引用**：报道来源、研究机构、引用的他人言论/观点
- **人物信息**：真实人物的姓名、职位、经历、言论是否准确
- **个人观点与经历**：区分「陈述事实」与「主观判断」，不将个人观点作为客观事实写入

**核查流程**：
1. **提取核查清单**：从素材中列出所有待验证的事实点（时间/数据/引用/人名/事件）
2. **搜索验证**：使用 `WebSearch` 工具对每一个核查点进行搜索验证，必要时多源对比
3. **时效性检查**：数据类内容须确认发布时间，优先使用最近 6 个月内的数据；超过 1 年的数据需说明或更新
4. **来源标注**：记录每个事实的验证来源（媒体/机构/官方渠道）
5. **标记风险项**：将「无法验证」「存在争议」「可能过时」的内容标记为风险项，在文章中进行模糊化处理或注明

**文章完成后输出筛查报告**，格式如下：

```
【事实核查报告】

✅ 已验证（N 项）
- [事实描述] | 来源：[媒体/机构] | 时间：[YYYY-MM]

⚠️ 风险项（N 项）
- [事实描述] | 问题：[无法验证/存在争议/数据过时] | 处理方式：[已模糊化/已删除/已注明]

❌ 已删除/修正（N 项）
- [原始描述] → [修正后描述] | 原因：[具体说明]

数据时效：本文所用数据截至 [YYYY-MM]（对话时间戳：[当前时间]）
```

**步骤3：文章创作**

以「真实故事计划」写作风格创作文章：

**写作风格核心特征**：
- **叙述视角**：第一人称为主，强调「我」的感受、困惑、选择
- **语言质地**：真实、克制、有质感，不刻意煽情，情绪在细节中渗透
- **叙事节奏**：慢进、不赶，细节密度高，给读者沉浸感
- **故事姿态**：不俯视，不仰视，与笔下的人同等高度站立
- **情感原则**：不说教，不定性，让读者自己感受
- **禁止词汇**：禁用「充满正能量」「逆袭成功」「感动了无数人」等廉价赞美
- **结尾处理**：开放式结尾，留有余地，让现实本身说话

**代表性叙事技法**：
- 开篇用一个极具现场感的细节/场景切入（不用「我叫XXX，今年XX岁」）
- 用具体动作代替情绪描述（「她把碗推到一边」而非「她很难过」）
- 穿插对话和内心独白，增加人物厚度
- 避免连续议论，让故事本身承载意义
- 文章标题：朴素有力，像一句陈述句或一个问题，不用感叹号

1. **生成文章结构**：
   - 字数：**2000～5000 字**（可根据素材量灵活调整）
   - 分 4～6 个自然段落，段与段之间有情节推进
   - 每个段落围绕一个时间节点或情绪转折

2. **撰写正文内容**：
   - 开篇：抛入具体场景（地点+时间+人物动作），3～5 句内建立现场感
   - 中段：故事的主要进程，穿插细节、对话、内心独白
   - 转折点：那个改变了一切的时刻，是全文情感密度最高处
   - 收尾：带着未解的余味，不强行给出答案

3. **优化润色**：
   - 检查是否有「流量写作」套路痕迹（如「扎心了」「破防了」「泪崩」）
   - 确认细节足够具体，避免泛泛而谈
   - 语言节奏是否有轻重缓急，避免每句话情绪密度相同
   - 检查结尾是否说得太透

4. **生成标题与文件名**（文章完成后）：
   - 基于全文内容，提炼核心叙事线索
   - 标题风格：**朴素、直白、有重量**，像一个陈述事实的句子
   - **文件名必须带时间戳**：格式为 `{标题}-{YYYYMMDDHHmm}`
     - 例如：`赶时间的人-202604171148`、`长江大桥上的守桥人-202604201530`
     - 时间戳取当前系统时间（精确到分钟）
     - **目的**：同一主题多次生成的文章不互相覆盖
   - 标题参考：
     - 《赶时间的人》（简洁，有多义性）
     - 《我在奥运会上做黄牛》（第一人称，具体职业/场景）
     - 《被资助长大的失足女孩》（人物状态，引发追问）
     - 《长江大桥上的守桥人》（具体地点+角色）
   - 生成 3～5 个备选标题，优先选最朴素的那个
   - 避免：感叹号、过度悬念词（「震惊」「绝了」「真相」）

**标题生成参考**：见 [references/story-topic-guide.md](references/story-topic-guide.md)

**步骤4：排版优化**

真实故事计划专属排版风格：

1. **整体基调**：
   - 克制、沉稳、留白充足
   - 不用醒目配色，主色调为深灰/黑色
   - 文字密度不过高，段落间有呼吸感

2. **结构设计**：
   - 不使用大量小标题分割（非干货文，避免碎片化）
   - 若需分段，用**数字序号**（一、二、三）或空行分隔
   - 金句用引用样式（`>` 块）突出

3. **细节处理**：
   - 对话文字适当缩进或引用
   - 重点信息可加粗，但不滥用
   - 不使用任何分隔线（`---` / `———`）

4. **结尾互动**：
   - 文章结尾加一段轻柔的引导语，不强迫互动
   - 示例：「如果你也有一个想被记录的故事，欢迎留言。」

**排版参考**：见 [references/formatting-best-practices.md](references/formatting-best-practices.md)

**步骤6：质量检查与最终交付**

1. **内容检查**：
   - 是否有猎奇感、消费苦难的倾向？若有，重写
   - 主人公是否是一个有立体感的人？而非符号化标签
   - 细节是否足够具体（时间、地点、对话、动作）？
   - 结尾是否留有余地？还是说教收尾？
   - 语言是否朴实有质感？有无廉价煽情词？

2. **事实核查结果输出**：
   - 输出完整的【事实核查报告】（见步骤2格式）
   - 报告置于文章交付之后单独呈现

3. **排版检查**：
   - 有无分隔线？（须删除）
   - 小标题数量是否过多导致碎片化？
   - 段落长度是否适中？

4. **HTML 检查**：
   - 在浏览器预览，排版是否正常？
   - HTML 文件大小是否合理（建议 <5MB）？

## 注意事项
- **不消费苦难**：故事中的困境是用来理解的，不是用来感叹或猎奇的
- **不替主人公下结论**：叙述者呈现，读者感受，不需要代替主人公总结人生
- **第一人称不等于自我宣泄**：克制是力量，避免情绪外泄
- **细节是灵魂**：「一个外卖员骑着车」不如「他骑着那辆前轮有点晃的电动车」
- 排版不使用任何分隔线
- **事实核查是强制流程**：每篇文章必须执行步骤2，不可因「内容看起来可信」而跳过
- **数据时效性底线**：引用数据发布时间超过 12 个月须重新搜索更新版本；超过 24 个月的数据不得引用，除非注明「历史数据」
- 充分利用语言理解能力，优先以叙事能力驱动，减少套路

## 使用示例

**示例1：职场主题**
- 用户输入：「我想写一个在工厂流水线工作的年轻人的故事」
- 执行：智能体以第一人称叙事，描绘流水线工人的一个具体工作日，抓住细节，呈现内心
- 关键处理：避免「励志」叙事，聚焦普通日子里的疲倦、小快乐、无解的困惑

**示例2：情感主题**
- 用户输入：「我想写一个离婚单亲妈妈的故事」
- 执行：从某一个具体的生活瞬间切入（比如独自签完离婚协议当天），展开一段真实的生活状态
- 关键处理：不把她处理成「坚强独立女性」符号，展现复杂的内心和选择

**示例3：边缘群体**
- 用户输入：「我想写老年痴呆患者的家属故事」
- 执行：以照料者（子女/配偶）第一人称叙述，记录照料日常中的某个时刻，以小见大
- 关键处理：不煽情，用细节传递爱与疲惫共存的真实状态
